# 对话与记忆系统设计

## 1. 目标

本文定义 3 件事：

1. 对话如何进入系统
2. 事件如何写入各角色记忆
3. agent 在决策时究竟能看到什么记忆输入

本文不定义新的显式动作接口。

记忆系统必须满足：
- 记忆是透明的，agent 不调用单独的“memory API”
- 记忆会持续影响判断、说话、结盟、报复和恐惧
- 同一事件对不同角色写入的记忆可以不同
- 公开事件、私聊事件、环境查询，三者写入方式不同
- 在 32 人规模下仍然可运行，不能无限堆积原始文本

## 2. 总原则

- “关系”不是独立数值系统，而是记忆中的长期判断和倾向
- agent 不直接读取数据库式状态，而是接收整理后的自身记忆输入
- 最近对话保留原始文本感，长期记忆保留压缩后的事实、判断和情绪残留
- 只有角色实际听到、看到、经历到的内容，才允许进入该角色记忆
- 环境 agent 的回答只进入发起查询者的私有记忆，不进入公共事件流
- 引擎负责记忆写入、压缩、归档和裁剪，agent 只负责使用结果

## 3. 对话对象

系统里的“可记忆输入”分为 4 类：

1. 公开对话
2. 定向对话
3. 环境查询往来
4. 公开制度事件

### 3.1 公开对话

来源：

```text
speak({ type: "public" }, text)
```

写入范围：
- 当前所有存活者

特点：
- 进入公共事件流
- 进入所有听者最近对话窗口
- 之后可能被压缩为长期记忆

### 3.2 定向对话

来源：

```text
speak({ type: "single", ids: [...] }, text)
speak({ type: "group", ids: [...] }, text)
```

写入范围：
- 发言者
- 实际接收者

特点：
- 不进入公共事件流
- 只进入相关角色最近对话窗口
- 可在后续被接收者压缩为“他对我说过什么”的记忆

### 3.3 环境查询往来

来源：

```text
speak({ type: "agent", id: "environment" }, text)
```

写入范围：
- 仅发起查询者

特点：
- 查询文本和环境回答都进入该角色最近对话窗口
- 只写入该角色私有记忆
- 不进入公共事件流
- 其他角色不会因环境查询自动得知内容

### 3.4 公开制度事件

来源示例：
- 广播 agent 宣读规则
- 系统宣布阶段开始
- 系统公布配对结果
- 系统公布淘汰结果
- 系统公布某轮公共结算信息

写入范围：
- 当前所有存活者

特点：
- 进入公共事件流
- 不一定以“对话”形式出现，但必须进入记忆候选

## 4. 最近对话窗口

### 4.1 定义

最近对话窗口是 agent 每次决策时都能直接看到的、较原始的短期上下文。

它保留：
- 最近发生的公开对话
- 最近发生的私聊或定向对话
- 最近的环境查询与环境回答
- 最近的广播和制度性文本

它不负责长期保存，只负责“让角色还记得刚刚发生了什么”。

### 4.2 数据结构

```text
conversation_window = [
  {
    event_id,
    turn_index,
    phase,
    speaker_id,
    target_scope,
    visibility,
    text,
    source_type
  }
]
```

字段说明：
- `event_id`：事件唯一标识
- `turn_index`：事件在本阶段内的顺序号
- `phase`：当前阶段
- `speaker_id`：发起者；环境回答可写为 `environment`
- `target_scope`：公开、单人、小组或环境查询
- `visibility`：`public` / `private` / `system`
- `text`：原始文本
- `source_type`：`speech` / `broadcast` / `environment_query` / `environment_reply` / `system_result`

### 4.3 长度限制

建议采用双限制：

- 条数上限：最近 `20~40` 条
- 文本上限：最近 `2,000~4,000` 字符

默认策略：
- 超上限时，从最旧记录开始裁剪
- 被裁剪的内容不直接消失，而是进入“记忆候选压缩流程”

### 4.4 对 agent 的意义

最近对话窗口的作用不是提供完整历史，而是保留：
- 目前场上最新的风向
- 刚刚谁说了什么
- 刚刚谁公开站队或翻脸
- 自己刚刚问过环境什么
- 自己刚刚答应过别人什么

## 5. 长期记忆

### 5.1 定义

长期记忆不是原始聊天记录，而是角色从经历中留下的较稳定痕迹。

长期记忆至少包含 4 层：

1. 事件记忆
2. 推断记忆
3. 情绪记忆
4. 倾向记忆

### 5.2 记忆项结构

```text
memory_item = {
  memory_id,
  round_number,
  phase,
  source_event_ids,
  memory_type,
  subject_ids,
  summary,
  belief,
  emotion,
  confidence,
  salience,
  visibility_scope,
  decay_tag
}
```

字段说明：
- `memory_id`：记忆唯一标识
- `round_number`：形成该记忆的轮次
- `phase`：形成该记忆的阶段
- `source_event_ids`：来源事件列表
- `memory_type`：记忆类型
- `subject_ids`：这条记忆主要涉及谁
- `summary`：压缩后的自然语言摘要
- `belief`：该角色当前对这条记忆的解释或判断
- `emotion`：对应的情绪残留，如好感、厌恶、亏欠、警惕、恐惧
- `confidence`：该角色对自身判断的主观确信度
- `salience`：当前重要性
- `visibility_scope`：`public_seen` / `private_heard` / `self_experienced` / `environment_only`
- `decay_tag`：衰减标签，用于后续裁剪

### 5.3 记忆类型

#### 事件记忆

记录发生过什么。

示例：
- “第 1 轮前，024 公开说不要信 004”
- “017 私下找我谈过合作”
- “我问环境现在谁还活着，环境回答只剩 8 人”

#### 推断记忆

记录角色从事件里得出的判断。

示例：
- “004 喜欢操控群体节奏”
- “011 可能在故意建立心理优势”
- “002 比他说出来的更在意女儿”

#### 情绪记忆

记录角色留下的主观感受。

示例：
- “我仍然对 015 的临时背约感到恼火”
- “020 的反应让我产生一点信任”
- “012 的说话方式让我不舒服”

#### 倾向记忆

记录较稳定的行为倾向和判断方向。

示例：
- “短期内不想公开得罪 004”
- “如果要找临时盟友，020 比较可能答应”
- “013 值得防备，不适合透露真实想法”

## 6. 记忆写入流程

每次事件发生后，系统按以下顺序处理：

1. 生成原始事件
2. 分发给有资格接收的角色
3. 写入这些角色的最近对话窗口
4. 判断该事件是否进入记忆候选
5. 对高显著性候选生成或更新长期记忆

### 6.1 原始事件结构

```text
event = {
  event_id,
  round_number,
  phase,
  source_type,
  speaker_id,
  target_scope,
  recipients,
  text,
  public_flag
}
```

### 6.2 记忆候选触发条件

以下情况默认进入记忆候选：

- 与生死、淘汰、通关直接相关
- 明确承诺、威胁、求助、结盟、指控
- 与角色底线、秘密、禁忌相关
- 语气或内容造成明显情绪波动
- 重复出现，形成模式
- 来自环境 agent 且直接影响当前决策
- 来自广播或系统，涉及规则变化或结果公布

普通闲聊不一定进入长期记忆，只保留在最近对话窗口中，等待自然淘汰。

### 6.3 显著性 `salience`

每条记忆候选都应计算显著性。

可以综合考虑：
- 是否涉及生死
- 是否涉及承诺与背叛
- 是否和自身动机直接相关
- 是否和自身秘密、底线相关
- 是否重复出现
- 是否刚刚发生
- 是否由公开场合发生

建议使用离散等级：

```text
salience = low | medium | high | critical
```

默认处理：
- `low`：通常只保留在最近对话窗口
- `medium`：可转为 1 条短期长期记忆
- `high`：必须形成长期记忆
- `critical`：必须形成长期记忆，并在后续输入中优先展示

### 6.4 新增与更新

记忆不一定每次都新增，也可以更新已有记忆。

示例：
- 原有记忆：“015 说过会跟我合作”
- 新事件：“015 关键时刻背约”
- 更新后记忆：
  - 事件记忆增加一条背约事实
  - 推断记忆更新为“015 承诺不稳定”
  - 情绪记忆增加恼火或羞辱感
  - 倾向记忆更新为“短期不再信任 015”

## 7. 记忆压缩

### 7.1 为什么要压缩

32 个角色长期运行时，原始事件量会迅速膨胀。

如果不压缩：
- agent 输入会越来越长
- 同类信息会反复出现
- 记忆会变成噪音堆，而不是决策材料

所以系统必须定期把零散事件归并成更少、更稳的记忆项。

### 7.2 压缩原则

- 保留对决策有用的信息，不保留所有原句
- 保留“谁、做了什么、对我意味着什么”
- 同类事件可以合并
- 被证伪的旧判断不删除，而是降级或改写

### 7.3 压缩示例

原始对话片段：

```text
015 说先一起走
015 说别担心
015 后来没有按约定行动
015 解释说自己也没办法
```

压缩结果：

```text
summary: "015 曾多次向我示好并口头承诺合作，但关键时刻没有兑现。"
belief: "015 在压力下优先自保，承诺可信度低。"
emotion: "恼火，戒备。"
```

### 7.4 压缩时机

建议在以下时机进行：

- 单阶段结束时
- 单轮结束时
- 最近对话窗口即将超限时
- 某角色记忆条目过多时

## 8. 遗忘与保留

记忆不应无限累积。

### 8.1 遗忘原则

- 与当前轮和当前生存压力无关的低显著性记忆逐渐淡化
- 反复被新事件强化的记忆更难消失
- 与底线、秘密、死亡、背叛、救助相关的记忆应长期保留

### 8.2 可采用的衰减标签

```text
decay_tag = ephemeral | short_term | stable | core
```

建议含义：
- `ephemeral`：极易遗忘
- `short_term`：数阶段后可降级
- `stable`：数轮内保留
- `core`：长期保留，几乎不主动删除

### 8.3 不应轻易删除的记忆

- 生死相关
- 明确背叛
- 明确救助
- 涉及自身核心动机
- 涉及自身底线
- 涉及自身秘密暴露风险

## 9. agent 实际可见的记忆输入

agent 在决策时，不看“全量数据库”，而看整理后的输入包。

建议结构：

```text
agent_input = {
  self_profile,
  conversation_window,
  memory_digest,
  global_info
}
```

其中：

### `self_profile`

包含：
- `background`
- `public_persona`
- `hidden_goal`
- `core_traits`
- `secret`
- `taboo`

### `conversation_window`

包含：
- 最近一段原始对话
- 最近广播和系统文本
- 自己与环境 agent 的最近往来

### `memory_digest`

不是全量记忆，而是当前最值得看的记忆摘要。

建议由以下几部分组成：
- 当前最重要的 `3~8` 条长期记忆
- 当前最相关的对象判断
- 当前最活跃的风险与机会
- 当前最需要兑现或提防的承诺

### `global_info`

包含：
- 当前轮次
- 当前阶段
- 当前规则
- 当前存活者
- 当前可用动作
- 当前倒计时或时间限制

## 10. `memory_digest` 建议结构

```text
memory_digest = {
  critical_memories: [...],
  active_promises: [...],
  active_threats: [...],
  suspicious_targets: [...],
  alliance_candidates: [...],
  emotional_triggers: [...]
}
```

说明：
- 这些是整理后的可见输入块，不是 agent 主动调用的接口
- 字段可以由引擎内部生成，但最终仍作为普通输入文本或结构化输入喂给 agent

## 11. 公开事件与私聊事件的差异

### 11.1 公开事件

特点：
- 多人同时听到
- 容易形成“公共叙事”
- 记忆里除了事件本身，还容易留下“别人也知道这件事”的判断

示例：
- “004 在所有人面前指控 010”
- “021 当众否认曾答应合作”

### 11.2 私聊事件

特点：
- 只对少数人可见
- 更容易形成秘密、承诺、试探和操控
- 记忆里除了事件本身，还会留下“这件事不能公开说”的判断

示例：
- “015 私下找我谈合作”
- “011 单独试探我是否愿意背刺别人”

### 11.3 环境查询

特点：
- 不是社交事件
- 但会影响角色判断
- 只进入发起者个人记忆

示例：
- “我刚确认了现在还剩 8 人”
- “我从环境回答里确认这轮规则没有第二次提交机会”

## 12. 承诺、背叛与验证

这 3 类内容应作为记忆系统重点支持对象。

### 12.1 承诺

当角色说出以下内容时，系统应倾向写入记忆候选：
- “我跟你走”
- “我不会投你”
- “这轮我保你”
- “先合作到这轮结束”

记忆中至少要保留：
- 谁向谁承诺
- 承诺发生在哪一轮哪一阶段
- 承诺内容大意

### 12.2 背叛

当后续行为和已有承诺明显矛盾时，系统应生成背叛型记忆。

示例：
- 先承诺合作，后公开切割
- 先承诺不投，后实际投票或下刀
- 先表示站队，后在关键动作中撤回支持

### 12.3 验证

不是所有判断都立刻定性。

系统应允许记忆经历：
- 初步怀疑
- 部分验证
- 彻底证伪
- 彻底证实

这样角色才会出现：
- 犹豫
- 误判
- 后悔
- 更强烈的报复或依赖

## 13. 引擎最低实现要求

如果只做 MVP，记忆系统至少要能做到：

- 正确区分公开、定向、环境查询三类输入
- 为每名角色维护自己的最近对话窗口
- 为每名角色维护自己的长期记忆列表
- 在阶段结束时进行一次记忆候选压缩
- 在生成 agent 输入时，输出 `conversation_window + memory_digest`
- 保证角色只能看到自己应看到的内容

## 14. 建议的 MVP 简化方案

如果第一版要优先跑通，可以先这样做：

### 最近对话窗口

- 固定保留最近 24 条

### 长期记忆

- 每名角色最多保留 30 条
- 超出后优先删除 `ephemeral`

### 记忆显著性

- 只用 `low / medium / high / critical`

### `memory_digest`

- 每次只提取最重要的 6 条

### 承诺与背叛

- 先用规则匹配和事件标注
- 不要求第一版就做复杂语义解析

## 15. 后续可扩展方向

- 更细的情绪标签
- 对同一事件的多视角偏差记忆
- 错误记忆与自我说服
- 长期创伤型记忆
- 更细的“谁知道谁知道”二阶记忆
