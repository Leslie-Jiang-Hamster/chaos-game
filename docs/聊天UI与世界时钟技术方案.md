# 聊天 UI 与世界时钟技术方案

## 1. 目标

将当前终端版同步交互，改造为“聊天软件式 UI + 世界时钟驱动”的多角色叙事系统。

玩家与所有 agent 在动作能力上完全对称。玩家只是 `001` 号角色的视角，不是系统特判对象。

## 2. 设计原则

- 所有角色共享同一套动作接口，不因目标是玩家还是 NPC 而变化。
- 世界时间独立推进，玩家不输入时，世界也持续运行。
- 前端只展示玩家可见的信息，不等于底层能力模型。
- 广播 agent 通过统一消息系统发言；环境信息以系统公共文本插入，不直接 `print`。
- 私聊是否可见由玩家视角决定，不由消息能力决定。

## 3. 统一动作模型

唯一基础动作：

```text
speak(target_scope, text)
```

目标范围：

```text
target_scope =
  { type: "public" }
  { type: "single", ids: ["014"] }
```

说明：

- 玩家私聊 NPC：`single -> ["014"]`
- NPC 私聊玩家：`single -> ["001"]`
- NPC 私聊 NPC：同样使用 `single`
- 广播 agent 只允许 `speak({ type: "public" }, text)`
- 不保留 `group`
- 不保留 `agent` 类型目标

## 4. 系统架构

### 4.1 世界状态层

- 维护全局 `world_time`
- 维护角色状态、消息流、会话列表、游戏阶段、规则、存活状态
- 负责推进世界 tick

### 4.2 调度层

- 每个角色有独立 `next_think_at`
- 到点后获得一次思考机会
- 先判断是否行动，再决定动作内容

### 4.3 动作执行层

- 负责消息分发
- 负责写入公共流、私聊流、广播流
- 负责触发记忆脏标记

### 4.4 记忆层

- 维护 `recent_window`
- 维护 `rolling_digest`
- 维护 `pinned_memory`
- 在角色真正思考前批量更新

### 4.5 展示层

- 以玩家视角渲染会话与消息
- 只展示玩家可见的消息

## 5. 世界时钟与调度

世界按固定小步长推进，例如：

- `tick = 300ms ~ 500ms`
- 每个角色的 `next_think_at` 带随机扰动

每次 tick：

1. 检查是否有系统事件到点
2. 检查哪些角色到达 `next_think_at`
3. 将这些角色放入思考队列
4. 逐个处理思考队列中的角色

系统事件包括：

- 广播
- 阶段切换
- 倒计时提醒

## 6. 角色思考流程

角色每次被唤醒，不直接生成发言，而是两阶段。

### 6.1 意图判定阶段

输入：

- 最近公共窗口
- 与自己相关的私聊窗口
- memory snapshot
- 当前阶段
- 当前规则
- 自己角色设定

输出：

```json
{
  "should_act": true,
  "channel": "public",
  "target_scope": { "type": "public" },
  "urgency": 0.72,
  "reason": "刚被点名，且公开场上话题与我强相关"
}
```

### 6.2 动作生成阶段

仅当 `should_act = true` 时调用：

```json
{
  "thought": "...",
  "action_name": "speak",
  "args": { ... }
}
```

这样可以减少无意义的大模型调用，并让“沉默”成为正常状态。

## 7. 消息系统

统一消息结构：

```text
Message
- id
- sender_id
- sender_name
- target_scope
- text
- source
- created_at
- visibility_key
```

`visibility_key` 用于前端路由：

- `public:lobby`
- `public:broadcast`
- `private:001-014`

消息分发规则：

- `public`：所有存活角色可见
- `single`：仅目标与发送者可见
- 不存在 `group`
- 不存在 `agent`

## 8. 前端 UI

采用聊天软件布局。

### 8.1 左栏：会话列表

- 公共大厅
- 广播
- 私聊对象列表
- 未读数
- 最近一句摘要

### 8.2 中栏：当前会话消息流

- 公共大厅显示所有公开消息
- 私聊窗口显示你与某一角色的单聊
- 广播窗口显示制度性消息
- 广播窗口也可以显示广播 agent 与玩家们的互动发言
- 关键环境描写以公共消息形式插入时间线

### 8.3 右栏：状态面板

- 当前阶段
- 当前规则摘要
- 存活人数
- 倒计时
- 玩家提交状态
- 可选：你的记忆锚点、可疑对象

### 8.4 底部输入框

- 当前会话决定发送目标
- 在公共大厅输入即 `public`
- 在某私聊窗口输入即 `single`
- 不存在环境窗口输入

## 9. 可见性与对称性

系统底层不区分：

- 给玩家发私聊
- 给 NPC 发私聊

统一都是：

```text
speak({ type: "single", ids: [...] }, text)
```

前端只显示玩家可见的私聊：

- 若 `001` 在参与者中，则展示
- 若 `001` 不在参与者中，则不展示

因此：

- 能力对称
- 可见性不对称
- 二者不冲突

## 10. 记忆系统

记忆分三层：

### 10.1 `recent_window`

- 最近公共消息
- 最近相关私聊
- 原文保留，用于短程上下文

### 10.2 `rolling_digest`

- 当前局势的主观印象
- 可压缩、可变化
- 用于短中程判断

### 10.3 `pinned_memory`

- 长期高价值锚点
- 不应轻易丢失
- 例如谁危险、谁可利用、谁说过关键话

更新策略：

- 新消息先进入 `pending_events`
- 不即时 digest
- 仅在角色下次真正思考前，批量 digest 一次
- 更新结果同时产出：
  - `rolling_digest`
  - `pinned_memory`

## 11. 发言自然性机制

避免“玩家一说话，固定两个人必须回复”。

规则：

- 角色是否发言取决于自己被唤醒时的判断
- 没有发言必要就沉默
- 发言人数不是固定值
- 公开发言存在冷却
- 私聊存在冷却
- 被点名、被质疑、被提问时，行动概率上升
- 阶段切换、广播后，行动概率短暂提升
- 不允许同一角色连续抢话过于频繁

## 12. 广播与环境

### 12.1 广播 agent

- 广播 agent 是公共时间线中的特殊角色
- 只能执行 `speak({ type: "public" }, text)`
- 既负责宣读规则、推进阶段、提醒时间，也可以像主持人一样与玩家互动
- 广播 agent 同样位于世界时钟内，受统一调度逻辑支配
- 广播窗口本质上是公共消息的一种筛选视图，而不是独立能力模型

### 12.2 环境描写

- 环境不再作为独立角色存在
- 不提供单独的环境问答会话
- 在关键时间点由系统向公共时间线植入固定环境文字
- 环境文字使用独立系统消息身份发送，不借用广播 agent 身份
- 这些文字用于建立空间感、压迫感、阶段变化和场面状态
- 环境描写不参与对话能力模型

## 13. 后端接口

建议后端使用 `FastAPI`。

### 13.1 `GET /api/bootstrap`

- 初始世界状态
- 初始会话列表
- 初始消息

### 13.2 `GET /api/state`

- 拉取增量消息
- 会话未读数
- 阶段变化
- 倒计时状态

### 13.3 `POST /api/send`

请求：

```json
{
  "conversation_id": "...",
  "text": "..."
}
```

由后端根据当前会话映射到统一 `speak(...)`

### 13.4 `POST /api/choose-number`

- 玩家提交数字

### 13.5 后端内部 scheduler

- 独立推进，不依赖前端请求触发 NPC 行为

## 14. 实时通信

MVP 建议：

- 第一版先轮询
- 前端每 `500ms ~ 1000ms` 请求一次增量状态
- 后端 scheduler 独立运行

后续可升级：

- WebSocket 推送消息与状态变化

## 15. 推荐技术选型

- 后端：`FastAPI`
- 前端：`React + Vite`
- 通信：先轮询，后续可切 WebSocket
- UI：单页应用

原因：

- 当前项目已是 Python，后端迁移成本低
- 聊天软件式 UI 更适合 React
- 后续加入多会话、未读、滚动、输入状态会更顺手

## 16. 第一版实现范围

- 公共大厅
- 广播会话
- 玩家与 NPC 私聊窗口
- 世界时钟 + 调度器
- NPC 自主公开发言
- NPC 可主动向任意单个对象私聊
- 前端只展示与玩家相关私聊
- 关键时间点的固定环境公共文本
- 第 1 轮规则与数字提交继续保留

第一版不做：

- NPC-NPC 私聊可视化
- 完整动画效果
- WebSocket
- 第 2 至第 5 轮

## 17. 改造步骤

1. 抽离游戏状态为后端服务对象
2. 增加 scheduler
3. 重构消息模型为会话系统
4. 接入前端 UI
5. 替换 CLI 主入口
6. 再优化自然发言策略

## 18. 待确认事项

1. 广播窗口是否作为公共大厅的筛选视图存在
2. NPC-NPC 私聊第一版是否允许存在但不展示
3. 第一版前端是否接受轮询而不是 WebSocket
