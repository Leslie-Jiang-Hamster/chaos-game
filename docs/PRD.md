# PRD: 混沌生还者

## 1. 产品定义

- 类型：单人、强叙事、多角色博弈、生存淘汰、AI Agent 模拟
- 核心形态：聊天软件式 UI + 世界时钟驱动的多角色互动系统
- 当前实现：Web MVP（标准库 HTTP + 单页前端），已跑通第 1 轮《诱饵均值》
- 目标赛制：5 轮，人数 `32 -> 16 -> 8 -> 4 -> 2 -> 1`
- 玩家身份：`001` 号参赛者视角，不是系统特判对象
- 胜负目标：玩家活到最后并独得奖金
- 失败条件：任意轮被淘汰死亡

## 2. 产品愿景

- 玩家不是在“点菜单过关”，而是在一个持续运转的残酷现场里活下去。
- 世界不会因玩家停顿而暂停，其他角色会继续交流、结盟、误导和背刺。
- 玩家与 NPC 在底层动作能力上保持对称，差异只来自视角与信息可见性。
- 核心乐趣来自读人、操纵预期、利用记忆、制造错觉，而不是解纯规则题。

## 3. 核心体验

- 规则本身不复杂，难点在读人、误导、结盟、反背刺。
- 每个角色都有身世、动机、底线和可利用点。
- 每轮游戏前后都有社交阶段，社交不是装饰，而是影响后续生死。
- 结果由规则、记忆、可见信息和临场判断共同决定。
- 玩家应始终感受到场面在流动，自己只是其中一名参与者。

## 4. 文本与呈现风格

- 整体文本基调偏细节描写、环境渲染和文学性表达，不写成功能手册。
- 场景文本应先建立陌生感、压迫感和空间感，再进入规则说明。
- 环境文字强调光线、气味、声音、空间压迫感、人物姿态等可感知细节。
- 广播 agent 的语言应简短、冷静、制度化，与环境描写和角色发言形成反差。
- 角色发言允许停顿、试探、含混、情绪和潜台词，不追求平均信息密度。
- 玩家看到的界面虽然是聊天 UI，但信息组织的第一优先级仍是叙事氛围。

## 5. 玩家视角与交互形态

### 5.1 玩家定位

- 玩家是 `001` 号角色，只拥有该角色应有的可见信息。
- 玩家与所有 NPC 共享同一套动作模型，不因为“主角”身份获得额外接口。
- 私聊是否显示，由玩家是否是参与者决定，不由动作能力决定。

### 5.2 目标交互形态

- 左栏：会话列表，包括公共大厅、广播、与各角色的私聊。
- 中栏：当前会话消息流，承载公开发言、私聊、广播和环境文本。
- 右栏：阶段、规则摘要、存活人数、倒计时、提交状态等世界状态。
- 底部输入框：玩家在当前会话中发送消息或执行当轮专属动作。

### 5.3 当前实现形态

- 当前默认入口是 Web 聊天界面，提供公共大厅、广播会话、私聊窗口和轮询同步。
- 当前仅保留 Web 聊天界面作为用户入口，不再维护其他交互形态。
- 玩家仍在第 1 轮通过发言与私聊互动，并秘密提交数字完成结算。

## 6. 统一动作与可见性模型

### 6.1 基础动作

唯一基础显式动作：

```text
speak(target_scope, text)
```

```text
target_scope =
  { type: "public" }
  { type: "single", ids: ["014"] }
```

约束：

- 玩家私聊 NPC、NPC 私聊玩家、NPC 私聊 NPC，统一使用 `single`。
- 广播 agent 只允许执行 `speak({ type: "public" }, text)`。
- 不保留 `group`。
- 不保留 `agent` 类型目标。
- “结盟”“威胁”“安抚”“套话”都是 `speak(...)` 的语义结果，不额外定义聊天动作。

### 6.2 可见性原则

- `public`：当前所有存活角色可见。
- `single`：仅发送者和目标对象可见。
- 底层能力对称，前端展示不对称。
- 若玩家 `001` 不在私聊参与者中，该私聊不会显示在前端。

## 7. 世界时钟与调度

### 7.1 世界时钟

- 世界时间独立推进，玩家不输入时，世界也持续运行。
- 建议固定 tick 推进，例如 `300ms ~ 500ms`。
- 阶段倒计时、广播、提醒、结算等都绑定到世界时间，而不是绑定到玩家输入。

### 7.2 角色调度

- 每名角色维护独立的 `next_think_at`，并带随机扰动。
- 角色到点后获得一次思考机会，先判断是否行动，再决定具体内容。
- 公共事件发生后，相关角色会获得记忆更新机会，但不要求立即发言。
- 不区分“前台角色”和“后台角色”，所有存活角色都进入统一调度。

### 7.3 系统事件

系统事件包括：

- 广播宣读规则
- 阶段切换
- 倒计时提醒
- 公共结算结果
- 固定环境文本注入

## 8. Agent 与思考流程

### 8.1 Agent 构成

- 31 名选手 agent
- 1 名广播 agent
- 环境文本由系统产生，不作为可交互 agent 存在

### 8.2 角色输入

每个选手 agent 仅可见：

- 自己的背景设定
- 自己的记忆快照
- 最近公共窗口
- 与自己相关的私聊窗口
- 可用动作

### 8.3 两阶段思考

第一阶段是意图判定：

```json
{
  "should_act": true,
  "reason": "刚被点名，且公开场上话题与我强相关"
}
```

第二阶段是动作生成，仅在 `should_act = true` 时调用：

```json
{
  "thought": "...",
  "action_name": "speak",
  "args": { ... }
}
```

这样可以：

- 让沉默成为正常状态
- 降低无意义的大模型调用
- 避免“玩家一发言就固定有人回”的机械感

## 9. 记忆系统

### 9.1 原则

- “关系”不是独立数值系统，而是记忆中的长期判断和倾向。
- 某人是否可信、是否可利用、是否令人恐惧、是否值得报复，统一保存在记忆里。
- agent 不显式调用记忆接口，而是在生成决策时自然使用记忆。
- 只有角色实际看到、听到、经历到的内容，才允许进入该角色记忆。

### 9.2 记忆分层

- `recent_window`：保留最近公共消息、最近相关私聊、最近广播与环境文本。
- `rolling_digest`：角色对当前局势的主观印象，可压缩、可变化。
- `pinned_memory`：长期高价值锚点，不应轻易丢失。

### 9.3 必须记住的信息

- 谁承诺过什么
- 谁骗过谁
- 谁救过谁
- 谁公开羞辱过谁
- 谁在关键轮投了谁
- 谁让谁产生好感、怀疑、厌恶或恐惧

### 9.4 更新策略

- 新消息先进入 `pending_events`。
- 不即时做 digest。
- 仅在角色下次真正思考前，批量更新记忆。
- 更新结果同时产出 `rolling_digest` 与 `pinned_memory`。

## 10. 消息系统

统一消息结构：

```text
Message
- id
- sender_id
- sender_name
- target_scope
- text
- source
- created_at
- visibility_key
```

`visibility_key` 用于前端路由：

- `public:lobby`
- `public:broadcast`
- `private:001-014`

消息分发规则：

- `public`：所有存活角色可见
- `single`：仅目标与发送者可见
- 广播是公共消息的一种特殊来源
- 环境文本以系统公共消息插入，不借用广播 agent 身份

## 11. 回合结构

每轮固定阶段：

1. 公共生活阶段
2. 赛前情报阶段
3. 游戏执行阶段
4. 结算淘汰阶段
5. 余波阶段

非游戏阶段重点：

- 了解彼此
- 交换信息
- 拉拢和误导
- 制造信任
- 埋下背叛

## 12. 五轮游戏

### 第 1 轮：诱饵均值

- 人数：`32 -> 16`
- 规则：
  - 所有人先公开发言
  - 然后各自秘密提交 `0-100` 的数字
  - 目标数 = 全体真实平均值的 `1/2`
  - 距离目标最近的 16 人存活
- 重点：公开共识可以被故意污染

### 第 2 轮：镜桥同行

- 人数：`16 -> 8`
- 规则：
  - 先配对
  - 每组共同走 6 段桥
  - 每段决定谁先走，再决定走哪边
  - 双方掌握不同桥面信息
  - 组内最终只活 1 人
- 重点：合作信息共享和转嫁风险同时存在

### 第 3 轮：四门筹码

- 人数：`8 -> 4`
- 规则：
  - 4 扇门，每扇门只活 1 人
  - 每人持 10 枚筹码，必须分配到 4 扇门
  - 每门最高者获胜
  - 一人最多赢 1 门
- 重点：口头分门协议和临场偷门背刺

### 第 4 轮：钥匙与刀

- 人数：`4 -> 2`
- 规则：
  - 每人秘密投出 1 把钥匙和 1 把刀
  - 钥匙代表保，刀代表杀
  - 分数高的 2 人存活
- 重点：纯拉票、纯追责

### 第 5 轮：终局审问

- 人数：`2 -> 1`
- 规则：
  - 双方先固定一个骰子点数 `1-6`
  - 进入审问轮
  - 每人每轮可提问，双方通过回答互相试探
  - 每人对自己给出的 4 个回答必须满足 `2 真 2 假`
  - 任一方可猜对方点数
  - 猜错直接死
  - 双方都不猜则下一轮继续，点数不变
- 重点：读人，不是算题

## 13. 技术架构总览

### 13.1 世界状态层

- 维护全局 `world_time`
- 维护角色状态、消息流、会话列表、游戏阶段、规则、存活状态
- 负责推进世界 tick

### 13.2 调度层

- 根据 `next_think_at` 唤醒角色
- 处理系统事件和角色思考队列
- 驱动广播、阶段切换和倒计时

### 13.3 动作执行层

- 负责消息分发
- 负责写入公共流、私聊流、广播流
- 负责触发记忆脏标记

### 13.4 记忆层

- 维护 `recent_window`
- 维护 `rolling_digest`
- 维护 `pinned_memory`
- 在角色真正思考前批量更新

### 13.5 展示层

- 以玩家视角渲染会话与消息
- 只展示玩家可见的信息
- 不暴露底层全部消息流

## 14. 接口与实现建议

### 14.1 推荐技术选型

- 后端：`FastAPI`
- 前端：`React + Vite`
- 通信：第一版轮询，后续可升级 WebSocket
- 形态：单页应用

### 14.2 建议接口

- `GET /api/bootstrap`
  - 返回初始世界状态、初始会话列表、初始消息
- `GET /api/state`
  - 返回增量消息、未读数、阶段变化、倒计时状态
- `POST /api/send`
  - 根据当前会话将前端输入映射到统一 `speak(...)`
- `POST /api/choose-number`
  - 处理第 1 轮数字提交

### 14.3 调度要求

- 后端 scheduler 必须独立推进，不依赖前端请求触发 NPC 行为。
- 第一版先使用轮询，前端每 `500ms ~ 1000ms` 拉取一次增量状态。

## 15. 范围划分

### 15.1 当前 Web MVP

- 载入 32 名角色数据
- 固定玩家开场设定
- 接入火山引擎 Responses API
- 使用缓存共享前缀提示词
- 环境引导与帮助提示
- 广播宣读规则前的自由社交
- 广播宣读规则后的第 1 轮执行
- 玩家通过 `speak` 与环境或角色互动
- 玩家秘密提交数字
- NPC 通过 LLM 生成发言、回复与数字
- 结算平均值、目标数、存活名单

### 15.2 第一版聊天 UI 目标范围

- 公共大厅
- 广播会话
- 玩家与 NPC 私聊窗口
- 世界时钟 + 调度器
- NPC 自主公开发言
- NPC 可主动向任意单个对象私聊
- 前端只展示与玩家相关私聊
- 关键时间点的固定环境公共文本
- 第 1 轮规则与数字提交继续保留

### 15.3 第一版暂不做

- NPC-NPC 私聊可视化
- 完整动画效果
- WebSocket
- 第 2 至第 5 轮的完整实现

## 16. 关键风险

- 角色说话同质化
- agent 使用不该知道的信息
- 社交阶段太空，实际无决策价值
- 游戏规则可算但不好玩
- 角色太多，玩家记不住
- 聊天 UI 若只做壳子，会削弱叙事压迫感
- 世界时钟若调度不稳，会出现节奏拖沓或消息风暴

## 17. 实施路线

1. 抽离游戏状态为后端服务对象。
2. 增加独立 scheduler 与世界时钟。
3. 重构消息模型为会话系统。
4. 接入聊天软件式前端 UI。
5. 持续演进 Web 主入口（服务框架、前端工程化与实时通信）。
6. 再优化自然发言策略、记忆批处理与实时通信。
